special-days 0.3.0 — what changed and why

A note to my future self. After running a deep, four-document review of the 0.2.x line, I implemented all of the recommendations on a single branch. The diff is large (33 files, +2150 / −1718 in the first commit, plus an audit pass) — this document explains why each piece moved.

The review documents themselves are at the repo root: review-concept.md, review-implementation.md, review-communication.md, review-development.md. This page is the implementation rationale; those are the diagnosis.

TL;DR. The package was trying to be both "always fresh" and "always offline" inside one runtime, and that ambition doubled the code surface. 0.3.0 collapses freshness into a CI release- bot, so the runtime is now a thin offline lookup over a JSON snapshot. Three confirmed bugs are fixed, one minor architectural smell is cleaned up, and the maintenance ritual moves almost entirely into CI.

Why the runtime Wikidata path is gone

This is the headline change. The 0.2.x design was a three-tier runtime lookup: shipped snapshot → per-user cache → Wikidata SPARQL on miss. About a third of the source and nearly half the tests existed to wire that chain together. The review's blunt finding was that it solved two problems where it could have solved one: "always offline" is the snapshot, "always fresh" is just "release often enough."

What I was trying to buy with the runtime fetch:

  • A user on an old version queries a newly-announced year and gets a real answer without upgrading.

What it actually cost:

  • Every API function carried an allow_network flag.
  • A new public exception (WikidataUnavailable) that callers had to either catch or explicitly opt out of.
  • A per-user cache file in $XDG_CACHE_HOME that became part of the package's correctness model — and which I verified could silently override the shipped snapshot with wrong data (BUG-3 in the review).
  • A race condition: two processes calling refresh() could interleave Path.write_text calls and produce a truncated cache file.
  • A user-agent that, if rate-limited, would degrade the package for every other user until the next __version__ bump.
  • Inability to run in firewalled / sandboxed environments without the user remembering to pass allow_network=False everywhere.

The Zen-of-Python complaint that finally tipped this for me: there were three ways to get a fresh answer (on-miss refresh, explicit refresh(), pip install --upgrade). Now there's one.

The replacement is a daily GitHub Actions job (refresh-snapshots.yml) that runs make snapshots-live, gates a PR on the live tests passing, and lets the existing OIDC release pipeline ship a patch version once merged. The user contract becomes: "if it's announced and you haven't upgraded in a week, upgrade." That's a contract a CI bot can keep without any user being clever.

_wikidata.py survived. It's still in src/special_days/ (so the build scripts and the live tests can import it via the regular package path), but nothing on the runtime lookup path touches it. The module-level comment says so explicitly; AGENTS.md repeats the rule.

Collapsing super_bowl.py / oscars.py into a generic Event

The 0.2.x version had two parallel modules whose only meaningful differences were:

  • The shipped snapshot filename.
  • The constant event name ("Super Bowl" vs "Academy Awards").
  • The edition-label function.
  • The is_X-named predicate.

Everything else — _load_snapshot, _cache_path, _fetch_from_wikidata, all_known, date, refresh, and the _Event-subclass scaffolding — was duplicated. Adding a third event (World Series, NCAA) would have cost another ~400 lines of copy-paste. The review pushed for a declarative registry; I landed it as a single Event value class:

_event = Event(
    name="Super Bowl",
    snapshot_resource=("special_days.data", "super_bowl.json"),
    edition_label=_edition_label,
)

def date(year):  return _event.first_date(year)
def dates(year): return _event.dates(year)
def all_known(): return _event.all_known()
def is_super_bowl_sunday(d): return _event.contains_date(d)

SuperBowl = _event.cls()

The class form is built dynamically by Event.cls(), which returns a fresh subclass of _EventDict with the parent Event attached as a class attribute. That way the registered subclasses still construct with the same zero-arg signature the old code had — SpecialDays(events=[SuperBowl]) still works exactly as before.

Adding the World Series will be ~20 lines plus the JSON snapshot, not 400 lines. The recipe is documented at how_it_works.md § Extending.

A snag along the way: don't name a method date inside the same class that uses date as a type

My first cut had Event.date(year) -> date. With from __future__ import annotations active, mypy resolves the return-type string lazily — and inside the class body date rebinds to the just-defined method, not the datetime.date import. Five "function not valid as a type" errors later, I renamed the method to first_date (the module-level function is still date(year), where the shadowing doesn't bite). Worth remembering: don't name methods the same as types in their own module.

Why the data shape is now {year: list[date]}

The trigger was the 3rd Academy Awards. The 2nd ceremony was held April 3, 1930; the 3rd on November 5, 1930. Under the old {year: date} shape, parse_event_results used out.setdefault(d.year, d) — first wins — so the November ceremony lost the year and never appeared in the snapshot or in any lookup result. The package was returning a smaller, lossier model of history than Wikidata actually contains.

The fix had to be structural. Three options:

  1. Keep {year: date} and synthesize a fake-year key (e.g. "1930.1") for the second ceremony. Ugly; spreads "is this key a year?" through every consumer.
  2. Key by edition number instead of year. Cleaner, but breaks the most common call (date(2025)) and forces every caller to track edition numbers.
  3. Change the data file to {"year": [date, ...]} and have date(year) return the first.

Option 3 is the only one that preserves the existing API for the 99% case and faithfully represents the data for the 1%. It also forward-compats events that recur within a year (rescheduled World Series Game 7 anyone?). The new {event}.dates(year) function returns the full list when callers need it.

The downstream consequences:

  • The snapshot files moved to the new shape; verified the wheel bundles them correctly.
  • parse_event_results now dedupes-and-sorts dates per year instead of setdefault-ing the first.
  • The edition labeller switched from year → label to date → label, which lets the 2nd and 3rd Oscars ceremonies in 1930 carry distinct labels.
  • The Oscars edition table is now keyed by exact date, which has the side benefit of fixing the off-by-one for 1931 (4th) and 1932 (5th) (review BUG-1).

Bugs fixed

All three confirmed bugs from the implementation review, plus the hazards.

BUG-1: Oscars edition labels off by one for 1931 & 1932

The 0.2.x code computed n = year - 1928, which quietly assumes exactly one ceremony per year. It breaks the moment two ceremonies share a calendar year (the 2nd and 3rd Awards, both in 1930). After that collision every label was shifted by one until the calendar resynced at the 6th Awards in 1934.

Fix: an explicit date-keyed table in oscars.py:

_OSCARS_EDITIONS = {
    date(1929, 5, 16): 1,
    date(1930, 4, 3):  2,
    date(1930, 11, 5): 3,
    date(1931, 11, 10): 4,
    date(1932, 11, 18): 5,
    date(1934, 3, 16): 6,
    # From 1934-03-16 forward, edition = year - 1928.
}

Falling back to year - 1928 for dates beyond the table keeps the existing formula working for the 90+ steady-state ceremonies. A regression test covers each of the early ceremonies plus the resync at 1934.

BUG-2: 3rd Academy Awards was invisible

Resolved structurally by the {year: list[date]} shape (see above). The snapshot now carries 1930-11-05; oscars.dates(1930) returns both ceremonies in chronological order.

BUG-3: cache silently overrides snapshot

Resolved by deleting the cache module entirely. I confirmed the bug in the review by writing {"2025": "1900-01-01"} to the cache file and watching super_bowl.date(2025) return 1900-01-01. There's no longer a cache file to poison.

HAZARD-1: non-atomic cache writes

Same — moot now that the cache is gone.

HAZARD-2: non-int year silently goes to the network

Fixed via _check_year, called at the top of Event.dates and Event.first_date:

def _check_year(year):
    if isinstance(year, bool) or not isinstance(year, int):
        raise TypeError(f"year must be int, got {type(year).__name__}")
    return year

The bool exclusion matters because isinstance(True, int) is True in Python; without it, super_bowl.date(True) would quietly mean super_bowl.date(1).

HAZARD-3: Roman numerals beyond XCIX

The old table only carried entries up to (100, "C"), so _roman(400) emitted CCCC instead of CD. The new table covers the full standard 1..3999 range. Property-style tests now cover every n in 1..399 known correct and every boundary (400, 500, 900, 1000, 3999). Out-of-range values raise ValueError.

HAZARD-4: datetime silently misses

The 0.2.x SuperBowl() instance was a dict subclass, and a datetime(2025, 2, 9) hashes differently from date(2025, 2, 9) — so membership tests returned False against a matching date. holidays.HolidayBase normalizes inputs; this package now does the same in _EventDict.__contains__/__getitem__/get/get_list and in LazyDateMap.

HAZARD-5: refresh() had no rate limiting

Moot — refresh() is gone.

Hazards tightened

SMELL-4: QID format validation

fetch_event_dates now matches Q[1-9]\d* before interpolating its argument into the SPARQL template. Today all callers pass hardcoded QIDs, but if the function ever grows caller-supplied input, the boundary is already guarded. Cheap defense in depth.

Bandit B310 / Ruff S310

urlopen on a non-pinned scheme is a Bandit/Ruff warning by default. The URL is always https://query.wikidata.org/sparql?query=… — there's no attacker-controllable scheme — so the warning is a false positive. Suppressed with a comment that explains why, not just that the line is exempt.

Cache silently surviving format drift

The dev review listed "the cache file format might drift from the snapshot file format" as a silent-rot risk. Both have moved to {year: list[date]}… and the cache no longer exists. One less invariant to police.

API surface removed

The following were on the 0.2.x public surface and are gone in 0.3.0. None of them existed long enough to have many third-party callers, but the CHANGELOG.md entry calls each one out in case someone hits an ImportError:

  • super_bowl.refresh() / oscars.refresh()
  • LazyDateMap.refresh()
  • SuperBowl(allow_network=…) constructor argument (and the same on Oscars, SpecialDays)
  • special_days.WikidataUnavailable from the public namespace (still present in special_days._wikidata for scripts and live tests)
  • special_days._cache module

What remains is exactly the value of the package:

  • date(year), dates(year), all_known(), and the is_X predicate per event module.
  • SuperBowl, Oscars, SpecialDays, LazyDateMap, union, EVENT_REGISTRY.

One thing that's new: dates(year) returns the per-year list (always [], length-1, or — for 1930 Oscars only — length-2). Useful when you want the full representation; date(year) still gives you the first.

Tooling and CI additions

AdditionWhy
refresh-snapshots.yml (daily cron) The load-bearing piece of the new architecture. Without this, the package goes stale by definition.
live-tests.yml (weekly cron) Catches Wikidata query drift even when the snapshot didn't change. Without it, a SPARQL regression would lie latent until the next push.
type-check CI job The package ships py.typed; downstream type- checked users rely on the package's annotations. Now mypy (default config) is gated. mypy --strict also passes — see [tool.mypy] strict = true in pyproject.toml.
.github/dependabot.yml Weekly bumps for GitHub Actions and pip dev-deps. Stops the slow rot of actions/checkout@v6-style version strings.
Origin-sync guard in _check_publish_ready git fetch origin main + assert HEAD == origin/main before allowing make publish-*. Prevents the "I forgot to pull" release from shipping.
type-check dependency in publish-* publish-{patch,minor,major} now depends on both test and type-check. Releases can't ship type errors.
Doctest of special_days._lazy in CI The union() docstring example was previously marked # doctest: +SKIP because it imported holidays. Rewritten to use a plain dict so it actually runs.
Release workflow artifact-action version match upload-artifact@v8 & download-artifact@v8 (was v7/v8; asymmetric majors will break eventually).

Documentation and metadata

The 0.2.x developer guide (docs/how_it_works-developer.html) was the strongest piece of writing in the project, but it was HTML, not rendered on GitHub, and not linked from the README. It's now docs/how_it_works.md, linked from the README and pyproject's Documentation URL.

Other doc moves:

  • README rewritten per the review-communication outline. The holidays-compat tagline now leads. The emoji-by-name composition pattern — previously hidden in examples/by_date.py — is inline. There's a section titled "Two ceremonies in one year" that walks the 1930 Oscars example so the new dates() function isn't a surprise.
  • New CHANGELOG.md with a full breaking-changes section for 0.3.0.
  • New CONTRIBUTING.md lifts the test / lint / release workflow out of the README.
  • New AGENTS.md orients LLM/coding agents to the package's conventions, lists suitable and unsuitable tasks, and points at how_it_works.md for the "extending" recipe.
  • New MAINTENANCE.md codifies the cadence-by-cadence playbook (daily / weekly / per-PR / per-tag).
  • New CITATION.cff for academic citation.
  • pyproject.toml: added holidays keyword, Libraries :: Python Modules classifier, and the full project-URL set (Documentation, Source, Changelog, Tracker).
  • Class docstrings: SuperBowl/Oscars are built from Event.cls() and now inherit a docstring that links the holidays package URL and explains the "calendar year of the event" semantics that has tripped people up on the Super Bowl (year 2025 = LIX = 2024 NFL season).
  • User-Agent fork warning in _wikidata.py: forks should change the URL so abuse complaints land in the right inbox.
  • KeyError messages now name the event and suggest upgrading the package.

Migration guide

Most callers don't need to change anything. The breaking surfaces:

0.2.x0.3.0
super_bowl.date(2099) may hit the network KeyError immediately (upgrade the package)
super_bowl.date(2025, allow_network=False) super_bowl.date(2025) (the only mode)
super_bowl.refresh() pip install --upgrade special-days
SuperBowl(allow_network=False) SuperBowl()
except WikidataUnavailable: No longer raised from the public API. If you import it from special_days._wikidata for the scripts, you can keep the handler.
oscars.date(year) when year had two ceremonies Same return (the earlier ceremony) — but oscars.dates(year) now exists if you want both.
Reading the per-user cache file directly The cache is gone; nothing to read.

The snapshot file format also changed ({"year": "date"} → {"year": ["date", ...]}). If anyone was reading those files directly (they shouldn't be — internal artifact), they'll need to follow the list shape.

What I'd watch next

  • Confirm the refresh bot actually runs on schedule. The first daily refresh after this merges should appear as a PR on branch bot/refresh-snapshots within 24 hours. If it doesn't, the workflow file likely needs workflow_dispatch permission tweaks or the contents: write permission isn't actually granted at the repo settings level.
  • The first Monday's live-test cron. Same — if it doesn't run, the schedule string or repo activity threshold is the culprit.
  • Dependabot's first wave of PRs. Probably 2-4 Action version bumps. Auto-merge candidate after CI green.
  • Python 3.10 EOL (Oct 2026). Drop it from the CI matrix and bump requires-python when it goes. Dependabot doesn't do this, so it's on me.
  • Adding the World Series. Now that the declarative Event abstraction is in, it's the right moment to validate it by adding one more event. The contract is "no copy-paste; ~20 lines plus a JSON snapshot." If the second addition needs more than that, the abstraction is wrong.
  • The "no runtime network" promise. Watch for accidental regressions where someone imports _wikidata from a runtime module. The simplest safeguard would be a CI grep: grep -RIn 'from .* import _wikidata' src/special_days/{super_bowl,oscars,_event,_lazy,__init__}.py must return empty.
