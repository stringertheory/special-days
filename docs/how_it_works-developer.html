What this package is (and isn't)

Given a year, return the date on which a special event occurred (or will occur). Currently ships the NFL Super Bowl and the Academy Awards (Oscars). The package is intentionally tiny:

  • Zero runtime dependencies. Only the Python standard library. urllib handles HTTP; json handles SPARQL results. Anything added to [project] dependencies in pyproject.toml is a long-term maintenance promise.
  • Offline-first. A snapshot of all known dates ships inside the wheel. Lookups for known years never touch the network.
  • Self-healing for future years. Asking for an unknown year transparently refreshes from Wikidata and caches the result.

It is not a general calendar/holidays library. The scope is deliberately narrow: a curated set of named, recurring events whose official date is announced once and rarely changes.

Public API surface

The package exposes two parallel APIs over the same underlying data. Pick whichever fits the question being asked.

  • Year-keyed (functional) — per-event module (special_days.super_bowl) with date(year) answering "when is THIS year's game?" Best for planners and schedulers.
  • Date-keyed (class) — SuperBowl, SpecialDays, LazyDateMap, all importable from special_days. Mirrors the holidays package's dict-like interface. Best for "is THIS date special?" composition.

Every event module exposes the same four-function shape:

FunctionBehaviorNetwork?
date(year, allow_network=True) Returns the datetime.date for the event in year. Raises KeyError if unknown and network is disallowed or also lacks the year. Yes, on miss (unless opted out).
is_super_bowl_sunday(d, allow_network=False) Predicate over a date. Default is local-only: historical "is this Super Bowl Sunday?" should never page over the network, and an honest "no" for an unknown future year is the right answer for almost all callers. No by default.
all_known() Returns a copy of everything we know offline (cache merged over snapshot). Mutating the result is safe and does not affect subsequent calls. No.
refresh() Force a Wikidata fetch and rewrite the cache. Propagates WikidataUnavailable on network/HTTP/JSON failure. Yes, always.

Date-keyed class API (holidays-compatible)

Every event also has a class form: SuperBowl in super_bowl.py, registered in special_days.EVENT_REGISTRY. The class is a dict[date, str] subclass that populates lazily per-year on first access — the same shape used by holidays.HolidayBase.

from datetime import date
from special_days import SuperBowl

sb = SuperBowl()                       # nothing loaded yet
date(2025, 2, 9) in sb                 # True; loads year 2025
sb[date(2025, 2, 9)]                   # 'Super Bowl'
sb.get_list(date(2025, 2, 9))          # ['Super Bowl']
list(sb)                               # [date(2025, 2, 9)] — only loaded keys

Three composition entry points layered on top:

SymbolUse case
SpecialDays(events=[...]) Lazy union of multiple registered events. Accepts string names, event classes, or pre-built instances. With no argument, includes everything in EVENT_REGISTRY.
union(a, b, ...) Lazy union of arbitrary date-keyed dict-likes. Use this to mix our events with holidays.HolidayBase instances, since the | operator on those may not play nicely across libraries.
LazyDateMap Read-only view returned by both of the above. Exposed primarily for type hints; users normally don't construct it directly.

Lazy semantics

Two rules govern all four lookup operations (__contains__, __getitem__, get, get_list):

  1. Year-by-year. A lookup at date(2025, ...) loads the entries for 2025 (if not already loaded) and only 2025. Each event keeps a set[int] of "years we've decided about" so a year with no announced date doesn't trigger repeated fetches.
  2. Source-by-source on union. LazyDateMap walks its sources in order and short-circuits at the first match for in / [] / get. get_list visits every source so multiple labels for the same date are all returned. Iteration deduplicates keys across sources.

Critically, for k in sd does not force a load — it iterates over the dict storage actually populated so far, which is just the years you've already touched (or what years=[...] preloaded). This mirrors holidays.US()'s behavior.

Labels

By default, the value for a hit is the constant event name ("Super Bowl"). This is deliberate: it makes name -> emoji-style flat-dict lookups work without year-specific special-casing. Pass label_with_edition=True on construction to get edition-numbered labels ("Super Bowl LIX") for display strings. Edition numbers are Roman; Super Bowl 50 (2016) is the documented Arabic-numeral exception.

Adding a new event class

Subclass _Event, give it a constant name, implement _date_lookup(year) (raising KeyError if unknown) and _upstream_refresh(), then register it in EVENT_REGISTRY in __init__.py. The dict-like surface, lazy population, refresh(), and years= constructor argument all come for free from the base class.

The three-tier lookup

Every lookup walks three sources in this order. Each tier is the authority for an answer the previous tier didn't have.

  1. Shipped snapshot. data/super_bowl.json, loaded via importlib.resources. Read once per call (cheap); always available, even with no filesystem write access and no network.
  2. Per-user cache. $XDG_CACHE_HOME/special-days/super_bowl.json (or ~/.cache/special-days/). Holds anything we've fetched fresh since the snapshot. Merged on top of the snapshot, so cache entries win on key collision.
  3. Wikidata SPARQL. Hit only when (1) ∪ (2) didn't have the year and allow_network=True. On success, the full fetched dataset is written to the cache and the requested year is returned.

Lifecycle of a date(year) call

Reading super_bowl.date() top-to-bottom:

  1. all_known() is built: snapshot loaded from importlib.resources.files("special_days.data") / "super_bowl.json", then _cache.read_cache(_cache_path()) is merged on top.
  2. If year in known, return known[year]. Done.
  3. If allow_network=False, raise KeyError(year).
  4. Otherwise call _fetch_from_wikidata(): _wikidata.fetch_super_bowl_dates() runs the SPARQL query, parses the result, and on a non-empty response writes the full {year: date} dict to the cache file.
  5. If the freshly fetched dict contains year, return it. Otherwise raise KeyError(year).

Two implicit guarantees worth noting:

  • The snapshot is always loaded fresh from disk on every call. There is no in-process memoization; this keeps test behavior simple and avoids state that survives across refresh() calls.
  • The cache write is all-or-nothing: an empty Wikidata response (e.g. transient parse failure) does not overwrite the cache with {}. See _fetch_from_wikidata's if fresh: guard.

The shipped snapshot

The snapshot is plain JSON, ISO dates keyed by stringified year:

{
  "1967": "1967-01-15",
  "1968": "1968-01-14",
  ...
  "2026": "2026-02-08"
}

It is bundled into the wheel via [tool.setuptools.package-data] in pyproject.toml:

[tool.setuptools.package-data]
special_days = ["data/*.json"]

Because of the src/ layout, “works for me in a checkout” and “works for a user who pip install special-days” are different questions. If package-data were misconfigured, local tests would still pass (the file is on disk) but installed users would get an empty snapshot. Running tests against the installed package (make install && make test) is the canonical check.

Two ways to regenerate

scripts/build_super_bowl_snapshot.py writes src/special_days/data/super_bowl.json from one of two sources:

  • make snapshot (no --live) → the EMBEDDED dict in the script. Hand-curated from NFL.com / Wikipedia. This is the source of truth for the shipped data. Use it when you want to ship a correction independent of what Wikidata currently says.
  • make snapshot-live → fetch from Wikidata. Use this before a release to pick up newly-scheduled future games. Diff the output against the embedded list before committing; if they disagree, decide whose answer to ship.

The on-disk cache

Same JSON shape as the snapshot. Location is $XDG_CACHE_HOME/special-days/<event>.json, falling back to ~/.cache/special-days/<event>.json.

The cache reader (_cache.read_cache) is deliberately permissive: any failure — missing file, malformed JSON, wrong top-level type, non-integer keys, non-ISO values — is treated as a cache miss and returns {}. The caller (the lookup pipeline) decides what to do next, which usually means “fall through to the next tier.” This is a load-bearing design choice: it means a corrupt cache file can never poison a lookup, only delay it by one network round-trip.

The cache writer atomically rewrites the whole file via Path.write_text after parents.mkdir(parents=True, exist_ok=True). It is not concurrency-safe across processes — two simultaneous refresh() calls could race — but for a single-user CLI/library use case this is fine.

The Wikidata SPARQL query

The query template lives in _wikidata.py as EVENT_DATES_QUERY:

SELECT ?item ?itemLabel ?date WHERE {
  { ?item wdt:P31  wd:{qid} . }    # instance of
  UNION
  { ?item wdt:P361 wd:{qid} . }    # part of
  UNION
  { ?item wdt:P179 wd:{qid} . }    # part of the series
  ?item p:P585 ?statement .
  ?statement psv:P585 ?dateValue .
  ?dateValue wikibase:timeValue ?date .
  ?dateValue wikibase:timePrecision ?precision .
  FILTER(?precision >= 11)       # day precision or finer
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en". }
}
ORDER BY ?date

The UNION over three predicates is intentional: Wikidata contributors use different relationship properties to attach an event instance to its series, and conventions drift over time. Matching all three gives the query the best chance of surviving a re-modeling.

The statement-level path (p:P585 → psv:P585) exists to read the wikibase:timePrecision qualifier. Wikidata stores P585 ("point in time") values with a precision: 11 = day, 10 = month, 9 = year. An upcoming event announced only down to the month (e.g. "February 2029") gets stored with precision 10 and a placeholder day of YYYY-MM-01. The FILTER(?precision >= 11) excludes those placeholders so they don't silently leak into results as bogus Feb-1 dates. A live test (test_live_wikidata.test_all_returned_dates_are_sundays) would catch the regression — every Super Bowl has been played on a Sunday, so a non-Sunday result on a date-precision query is a strong signal a precision-10 placeholder slipped through.

The HTTP layer (sparql_query) does only what's strictly necessary:

  • GETs https://query.wikidata.org/sparql?query=... with Accept: application/sparql-results+json.
  • Sends a polite User-Agent identifying the package and version, as Wikimedia's policy requires.
  • Wraps every error — HTTPError, URLError, TimeoutError, decode failures, non-JSON bodies — in a single WikidataUnavailable exception. The lookup pipeline catches nothing; it lets the exception propagate to refresh() callers.

The response is parsed by parse_event_results, which walks the standard SPARQL 1.1 Query Results JSON Format shape:

results.bindings[*].date.value  →  "2025-02-09T00:00:00Z"

The Z suffix is rewritten to +00:00 so datetime.fromisoformat on pre-3.11 interpreters accepts it. Bindings missing a date are skipped; unparseable dates are skipped. When two events of the same series fall in the same calendar year (rare but possible — think rescheduled World Series), the earlier one wins via setdefault, so results are deterministic across runs.

Network policy and failure modes

ScenarioBehavior
date(known_year) Never touches the network. Returns from snapshot/cache.
date(unknown_year, allow_network=False) Raises KeyError. No network attempt.
date(unknown_year), Wikidata online, year present One SPARQL request. Cache rewritten with the full fetched set. Returns the date.
date(unknown_year), Wikidata online, year absent One SPARQL request. Cache rewritten. Raises KeyError.
date(unknown_year), Wikidata unavailable WikidataUnavailable propagates from refresh(); for date() it is not caught, so callers see it.
is_super_bowl_sunday(d) Never touches the network unless allow_network=True is passed explicitly. KeyError is caught and converted to False.
refresh() Always one SPARQL request. Returns the freshly-fetched dict. WikidataUnavailable on any failure.

Maintenance: regenerating the snapshot

Two situations trigger a snapshot refresh:

  1. A new game gets officially announced. Add the date to the EMBEDDED dict at the top of scripts/build_super_bowl_snapshot.py, then run make snapshot to regenerate the JSON. Commit both.
  2. You want to verify against Wikidata before a release. Run make snapshot-live, then git diff src/special_days/data/super_bowl.json. If the diff is empty, Wikidata agrees with the embedded list. If it shows differences, investigate before committing.

If the live tests start failing (SPECIAL_DAYS_LIVE_TESTS=1 make test-live), Wikidata has probably reshaped its data model for that event. The first place to look is EVENT_DATES_QUERY in _wikidata.py. After updating, cut a new release so users of the network fallback get the fixed query.

Extending: adding a new event type

To add, say, the Oscars:

  1. Find the Wikidata Q-ID for the series (e.g., the Academy Awards is Q19020; verify on wikidata.org).
  2. Create src/special_days/oscars.py mirroring super_bowl.py:
    • Functional surface: date(year), an is_oscars_night-style predicate, all_known(), refresh(), plus the internal _SNAPSHOT_RESOURCE / _CACHE_FILENAME constants.
    • Class surface: an Oscars(_Event) class with a constant name attribute and _date_lookup / _upstream_refresh / (optionally) _label_for overrides.
  3. Add a thin wrapper in _wikidata.py: fetch_oscars_dates() just calls fetch_event_dates("Q19020"). The SPARQL query template is already general.
  4. Build a snapshot: extend scripts/build_super_bowl_snapshot.py into a similar build_oscars_snapshot.py (or refactor the script if you're adding more than one event at a time), with an EMBEDDED dict of hand-verified dates.
  5. Add data/oscars.json by running the script. The package-data glob (data/*.json) already picks it up.
  6. Register the class in __init__.py: add "oscars": Oscars to EVENT_REGISTRY and re-export Oscars. This makes SpecialDays(events=["oscars"]) and SpecialDays() (defaulting to all events) work.
  7. Add tests mirroring the Super Bowl tests, including a live test that verifies the SPARQL query against the real endpoint.

Don't generalize prematurely beyond _Event. Each event module is small; the cost of light duplication in super_bowl.py-shaped files is lower than the cost of building elaborate per-event abstractions before they're warranted.

Testing model

The test suite is intentionally layered:

  • Mocked unit tests (test_super_bowl.py, test_wikidata.py, test_cache.py, test_collection.py) run on every invocation of make test. They mock at clear boundaries — special_days._wikidata.urlopen for the HTTP layer, and special_days.super_bowl._fetch_from_wikidata for the public-API tests. Tests do not reach across modules to mock. test_collection.py covers LazyDateMap, union, and the SpecialDays registry, including an end-to-end interop test that simulates a holidays.HolidayBase source.
  • Live Wikidata tests (test_live_wikidata.py) are gated on SPECIAL_DAYS_LIVE_TESTS=1 and skip by default. Their job is to detect query drift: if Wikidata's modeling of the Super Bowl changes such that our SPARQL query no longer returns the right items, these tests fail loudly. Run them before cutting a release.

Sample test invocations:

# everything fast (default)
make test

# one class
.venv/bin/python -m unittest tests.test_super_bowl.UnknownYearTests -v

# one method
.venv/bin/python -m unittest \
  tests.test_super_bowl.UnknownYearTests.test_raises_when_network_also_lacks_year

# include live Wikidata
make test-live
